# ブレインダンプアプリ DB仕様書

## 1. 概要

- データベースは Supabase PostgreSQL を利用する。
- テーブルは **`categories`（カテゴリマスタ）** と **`brain_dumps`（メモ）** の2つとする。
- カテゴリの選択肢はマスタテーブルで管理し、メモは `category_id` で参照する（未選択時は `NULL`）。
- 現行の `index.html` では RLS を使わず、anon key で CRUD する（将来は RLS・認証を追加予定）。

## 2. テーブル定義

### 2.1 `categories`（カテゴリマスタ）

ドロップダウンの選択肢を管理する専用テーブル。

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PRIMARY KEY | カテゴリの識別子 |
| name | text | NOT NULL / UNIQUE | カテゴリ名（一覧・ドロップダウン表示用） |
| sort_order | integer | NOT NULL / DEFAULT 0 | 表示順（小さいほど上） |
| created_at | timestamptz | DEFAULT now() | 登録日時 |

### 2.2 `brain_dumps`（メモ）

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PRIMARY KEY | レコードの識別子 |
| title | text | - | タイトル |
| category_id | uuid | FOREIGN KEY → `categories.id` / NULL 可 | カテゴリ（任意） |
| content | text | - | 本文 |
| next_action | text | - | 次に取る行動 |
| created_at | timestamptz | DEFAULT now() | 作成日時（未指定時は現在日時） |

**外部キー**

- `category_id` → `categories(id)`
- 削除時: `ON DELETE SET NULL`（カテゴリ削除時、メモの `category_id` は `NULL` になる）

### 2.3 カラム詳細（`brain_dumps`）

| カラム名 | 補足 |
|---|---|
| id | UUID。主キー。新規登録時は DB で自動採番（`gen_random_uuid()`） |
| title | アプリ側では必須入力として扱う |
| category_id | **任意（NULL 可）**。未選択時は `null`。一覧では `categories.name` を結合して表示 |
| content | アプリ側では必須入力として扱う。URL はプレーンテキストとして保存し、一覧表示時にクライアント側でリンク化する |
| next_action | 任意。未入力時は `null`。URL はプレーンテキストとして保存し、一覧表示時にクライアント側でリンク化する |
| created_at | insert 時にアプリから送らず、DB デフォルトを利用する |

### 2.4 論理上の列順

**`categories`**

`id` → `name` → `sort_order` → `created_at`

**`brain_dumps`**

`id` → `title` → `category_id` → `content` → `next_action` → `created_at`

### 2.5 テーブル関係（概要）

```
categories (1) ──< (0..n) brain_dumps
     id              category_id (NULL 可)
```

## 3. DDL（Supabase/PostgreSQL 想定）

実行用 SQL は `docs/20260523_01_カテゴリマスタ追加.sql` を正とする（マスタ作成・メモテーブル・移行・仮データを1ファイルに含む）。

### 3.1 新規作成（全体）

```sql
create extension if not exists pgcrypto;

create table public.categories (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  sort_order integer not null default 0,
  created_at timestamptz default now(),
  constraint categories_name_unique unique (name)
);

create table public.brain_dumps (
  id uuid primary key default gen_random_uuid(),
  title text,
  category_id uuid references public.categories(id) on delete set null,
  content text,
  next_action text,
  created_at timestamptz default now()
);
```

### 3.2 既存環境への移行

以前 `brain_dumps.category`（text 型）を使っていた場合の手順。

1. `categories` テーブルを作成
2. カテゴリ仮データを投入
3. `brain_dumps` に `category_id` を追加（NULL 可）
4. 旧列 `category`（text）を削除

```sql
alter table public.brain_dumps
  add column if not exists category_id uuid
  references public.categories(id) on delete set null;

alter table public.brain_dumps drop column if exists category;
```

### 3.3 既存テーブルの NOT NULL 解除

```sql
alter table public.brain_dumps alter column title drop not null;
alter table public.brain_dumps alter column content drop not null;
```

## 4. インデックス方針

- 主キーインデックス: `id`（各テーブルで自動作成）
- `brain_dumps.created_at` … 一覧の降順ソート用（必要に応じて追加）
- `brain_dumps.category_id` … カテゴリ結合・一覧のカテゴリ絞り込み用（現行はクライアント側フィルタ。DB側 where 絞り込みに拡張する場合は追加を検討）

## 5. セキュリティ設定（将来）

現行アプリは RLS 無効で運用する。将来、要求仕様どおりユーザーごとにデータを分離する場合は次を検討する。

- 所有者カラム（例: `user_id uuid`）の追加
- `brain_dumps` / `categories` で RLS を有効化
- 認証済みユーザー向けの `select` / `insert` / `update` / `delete` ポリシー

## 6. 運用ルール

- アプリケーションは Supabase SDK 経由でテーブルを操作する。
- ドロップダウン用のカテゴリ一覧は `categories` を `sort_order` 昇順で取得する。
- メモ保存時、`category_id` は任意。未選択時は `null` を送る（空文字は送らない）。
- 一覧表示では `brain_dumps` と `categories` を結合し、カテゴリ名（`categories.name`）を表示する。未設定時は「—」などと表示する。
- カテゴリ絞り込みはアプリ側で取得済みメモを `category_id` でフィルタする（スキーマ変更は不要）。
- `content` / `next_action` に含まれる URL のリンク表示はアプリ側の表示処理のみとし、DB・列定義の変更は行わない。
- `id`（メモ・カテゴリとも）は DB で自動生成し、アプリ側で採番しない（カテゴリマスタの手動追加時を除く）。
- `created_at` は DB デフォルト値を利用し、メモの insert 時にアプリから明示指定しない。
- `next_action` は任意。未入力時は `null` を送る。
